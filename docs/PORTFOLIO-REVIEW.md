## 1. 백엔드 리뷰 (현업 관점)

### 강점 — 이력서·면접에서 그대로 팔리는 것들
1. **실서비스 운영과 인시던트 이력**: 배포·모니터링·장애대응·포스트모템(`TROUBLESHOOTING.md` 18건)이 있는 신입 포트폴리오는 극소수. 특히 #18(Grafana 크래시루프)은 타임라인·근본원인 2개·교훈까지 갖춘 진짜 인시던트 문서다.
2. **트랜잭션 경계에 대한 실전 이해**: self-invocation으로 죽어있던 알림(#96), 트랜잭션 밖 비관적 락으로 매일 죽던 배치(#178), 크롤링 I/O를 트랜잭션 밖으로 분리한 `CollectPersistenceService` — 전부 면접 단골 주제의 실전판.
3. **돈이 걸린 버그의 수습**: 재구독 UNIQUE 위반으로 "결제만 되고 구독 실패"(#179) — 외부 API는 롤백되지 않는다는 것을 검증 순서로 해결. 결제 도메인 경험으로 어필 가능.
4. **성능 개선의 실제 사례**: N+1 → IN 배치(#98), 가격 비정규화 컬럼(#100, 윈도우 함수 정렬 제거), 페이지네이션 네이티브 쿼리.
5. **점진적으로 성숙해진 보안**: admin 노출 차단(nginx), HttpOnly 쿠키 전환(#185), rate limiting, 빌링키 AES 암호화.
6. Flyway V13까지의 마이그레이션 이력, 도메인 패키지 구조, 예외 코드 체계(`ErrorCode`) — 기본기가 정돈됨.

### 약점 — 현업 리뷰어가 지적할 것들 (우선순위 높은 순)
1. **통합 테스트 0** (실측): 모든 테스트가 Mockito 단위/WebMvcTest 슬라이스. "리포지토리 네이티브 쿼리(윈도우 함수)가 진짜 MySQL에서 도는지"를 검증하는 테스트가 없다 — 실제로 V13 백필을 수동 MySQL 컨테이너로 검증했던 것이 그 공백의 증거.
2. **API 응답 형태 불일치** (실측): `Map<String,String>`, raw String("수집 완료"), DTO, Page가 혼재. 공통 래퍼/에러 스키마 부재는 과제전형 채점 기준에서도 자주 보는 감점 포인트.
3. **JWT 필터 매 요청 DB 조회** (#97, 열려있음): 인증이 트래픽과 함께 DB 병목이 되는 구조.
4. **랭킹 전체 메모리 조회** (#99, 열려있음): 455건이라 지금은 문제없지만, "왜 이렇게 뒀는지"를 물으면 측정 데이터로 답할 수 있어야 함.
5. **시크릿 기본값**: `BILLING_ENCRYPTION_KEY`에 0x00... 기본값이 코드에 존재 — 운영에선 덮이지만, 리뷰어는 "기본값이 있으면 언젠가 그걸로 뜬다"고 지적한다.
6. **단일 인스턴스 전제가 암묵적**: findOrCreate 레이스, 스케줄러 중복 실행 방지(ShedLock류) 없음. 전제 자체는 MVP로 타당하나 **코드/문서에 전제를 명시**해야 방어가 된다.
7. 크롤러 견고성: 파서 버전·이상탐지는 있으나, HTML 구조 변경 시 알림→수동 대응 흐름이 문서화만 되어 있음.

## 2. 프론트 리뷰 (현업 관점)

### 강점
1. Jest 137개 전부 통과, 컴포넌트·훅·API 레이어 테스트가 고르게 있음 (신입 프론트 평균 이상).
2. HttpOnly 쿠키 전환을 백엔드와 계약 맞춰 완주 — 401→refresh 단일 비행, 레거시 토큰 마이그레이션까지.
3. SEO 실전(동적 sitemap 461 URL, OG, force-dynamic 함정 해결), STOMP 실시간 알림, App Router + SWR 패턴 일관성.

### 약점
1. **API 응답 런타임 검증 없음** (실측): 응답을 타입 캐스팅만 함. **이 구조 때문에 실제 사고가 2번 났다** (`sourceUrl` vs `url` 필드 불일치, 상세 페이지 `latestPrice` 부재 — 둘 다 타입체크를 통과하고 운영에서 조용히 깨짐).
2. **E2E 테스트 0**: 로그인→찜→알림 같은 핵심 여정의 스모크가 없어서, 배포 검증을 매번 수동 curl로 함.
3. 검색 디바운스 부재(fe#48), 번들/Lighthouse 측정 이력 없음.

## 3. 취업 관점 개선 로드맵 (우선순위순)

각 항목의 가치는 "구현 난이도"가 아니라 **면접에서 만들어지는 이야기**로 평가했다.

### P1. Testcontainers 통합 테스트 도입 — 가장 취업 직결
- 대상: `PriceHistoryRepository`의 윈도우 함수 쿼리 3~4개, Flyway 마이그레이션 전체 적용 검증, `CollectPersistenceService` 트랜잭션 커밋 경계
- 면접 스토리: "슬라이스 테스트만으론 네이티브 쿼리를 못 믿어서 Testcontainers로 실제 MySQL 8을 띄웠다" — 과제전형에서 테스트 작성 근육과 직결되는 유일한 항목
- 완료 기준: `@DataJpaTest`+Testcontainers 프로파일, CI에서 실행

### P2. #97 JWT 필터 DB 조회 제거 — "측정→개선→측정" 스토리 만들기
- **개선 전 반드시 측정부터**: 요청당 쿼리 수/응답시간을 기록해두고, claim 기반 인증으로 바꾼 뒤 재측정
- 면접 스토리: "왜 DB 조회를 했었나 → 즉시 권한 반영 때문 → 제거하면 어떤 보안 문제(탈퇴/정지 회원)가 생기나 → 어떻게 보완했나" — 트레이드오프 문답 그 자체
- 주의: 정답을 서두르지 말 것. 이 항목은 결과보다 **의사결정 기록**이 산출물

### P3. API 응답 표준화 + 에러 스키마 문서화
- 공통 래퍼(또는 최소한 에러 응답 통일) + springdoc으로 OpenAPI 완성 (springdoc 의존성은 이미 있음)
- 면접 스토리: "혼재된 응답 형태가 프론트 사고 2건의 원인이었다"로 시작하면 완벽한 동기 부여 서사

### P4. 프론트 응답 런타임 검증 (zod) — 실사고 2건이 동기인 완벽한 스토리
- `lib/api/*`의 캐스팅을 zod 스키마 파싱으로 교체, 실패 시 명시적 에러
- 면접 스토리: "타입스크립트는 컴파일 타임만 지켜준다 — 실제로 백엔드 필드명 불일치가 운영에서 조용히 깨졌고, 계약을 런타임에 강제하도록 바꿨다"

### P5. Playwright E2E 스모크 3개
- 홈 로드, 종목 상세(가격 표시), 로그인 여정. CI에 연결하면 "배포 검증 자동화" 서사 완성

### P6. 부하 테스트 1회 (k6) — 신입 차별화 끝판
- 목록/상세/차트 API에 대해 "몇 RPS에서 무엇이 먼저 무너지는가" 측정 → #99(랭킹 메모리)가 실제 병목인지 데이터로 확인 → 개선 → 재측정
- 면접에서 "제 서비스는 X RPS까지 버티고, 병목은 Y였습니다"라고 말하는 신입은 없다

### P7. README를 채용 담당자용으로 재구성
